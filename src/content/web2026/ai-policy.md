## What's cool and what's not

I get it. AI tools are everywhere now and yes they're pretty useful for certain things. I am not an unreasonable caveman who wants to prevent you from making your life easier, but there are also some rules on how I expect you to engage in this class creatively. 

## What's Fine

**Code stuff**: AI is great for the technical heavy lifting. Got stuck on a CSS issue? An LLM can help debug that. Want to understand how a particular tool or command works? AI can explain it patiently for the most part.

**Learning support**: Use AI to clarify concepts, research design patterns, get tool recommendations, or troubleshoot technical problems. If you're stuck on stuff like "how do responsive grids work?", it's okay with me if AI is your friend.

## What's Not Fine

**Any written work**: Essays, reflections, design rationales, project descriptions, your manifesto, your reply letter. Nope. These need to be in your voice, with your thoughts. I want to read what you think, not what GPT thinks you should think. You ain't bothering to write it, I ain't bothering to read it.

**Creative work**: Don't use AI to generate your graphics, illustrations, please. Same goes for website copy or creative writing. That may not be your preference, but please try to stick with this for the duration of the class. The whole point is for you to develop your own design voice and perspective.

## Asking better questions

Knowing *how to talk to an LLM about code* is a real skill, and it's part of this course. When you ask an AI (or me, or a neighbor) for help, the quality of the answer is capped by the quality of the question. The pattern:

1. **Show, don't summarize.** Paste the actual error message and the actual code, not "my flexbox isn't working." The error text is the single most useful thing you have.
2. **Make the smallest broken thing.** If your whole page is misbehaving, copy the misbehaving part into a tiny separate file and reproduce the problem there. Half the time you'll find the bug yourself while doing this. (This is called a *minimal reproduction* and professionals do it constantly.)
3. **Say what you expected vs. what happened.** "I expected the image to sit next to the text; instead it's below it and stretched."
4. **Ask for an explanation, not just a fix.** End with "explain why this happens", otherwise you'll be back tomorrow with the same bug wearing a different hat. If the AI gives you code, make sure you can explain every line of it. If you can't, ask it to explain until you can.
5. **Know where the LLM sits in the stuck protocol.** It's step five: read the error aloud → rubber duck → docs → neighbor → me or the LLM. The steps before it aren't bureaucracy; they're where the learning happens.

## The Deal

If you use AI for the acceptable stuff, just tell me about it. Something like: "I used ChatGPT to help debug my code and generate the initial HTML structure for my navigation." That's it. No big deal, just be transparent. You must:

- Clearly state what AI tool was used (ChatGPT, Claude, Copilot, etc.)
- Specify exactly how it was used (debugging, code generation, concept explanation)
- Include this declaration in your submission or project documentation

**Example Declaration:**

> "I used Claude to help debug my CSS flexbox layout and to generate the initial HTML structure for my navigation component."

---

**If still uncertain, ask me before submission.** The bottom line: AI should help you learn and build, not think and create for you. There's a difference, and I trust you to know it.
